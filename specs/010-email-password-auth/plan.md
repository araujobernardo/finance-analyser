# Implementation Plan: Email and Password Sign-Up and Sign-In

**Branch**: `010-email-password-auth` | **Date**: 2026-05-06 | **Spec**: [spec.md](spec.md)

---

## Summary

Implement a full email-and-password authentication system on top of the existing Express server (feature 009) and PostgreSQL/Drizzle layer (feature 008). Adds 5 new columns to the `users` table via a Drizzle migration, six Express routes under `/api/auth/`, two server-side service modules (auth + email), and four React frontend pages. Migrates the existing tab-based App.tsx to URL-based React Router routes and adds a `ProtectedRoute` guard that redirects unauthenticated users to `/login`.

---

## Technical Context

**Language/Version**: TypeScript 5.9 (strict mode)
**Primary Dependencies**: `bcrypt` (cost 12), `jsonwebtoken` (15min access token, HS256), `resend` (email delivery), `react-router-dom` v7 (already installed)
**Storage**: Supabase PostgreSQL via Drizzle ORM — existing `users` table extended with 5 new columns; migration `0001_auth_tokens.sql`
**Testing**: Vitest (existing) + `tsc -p tsconfig.server.json --noEmit` + `tsc -b`
**Target Platform**: Node.js (server, Railway) + browser (React, Vite)
**Project Type**: Web application (existing Express server + React frontend)
**Performance Goals**: Registration completes in <2min (SC-001); sign-in in <30s (SC-002); password reset flow in <5min (SC-003)
**Constraints**: Token hashing via SHA-256 (Node built-in `crypto`); JWT stored in `sessionStorage`; no token refresh; no social login; no 2FA; email addresses normalised to lowercase before storage and comparison

### Task 6 — Create `src/server/services/authService.ts`

Central service for all cryptographic operations. No external packages beyond those installed in Task 1 and Node.js built-ins.

```typescript
import bcrypt from "bcrypt";
import jwt from "jsonwebtoken";
import crypto from "crypto";

const BCRYPT_COST = 12;
const JWT_EXPIRY = "15m";
const TOKEN_EXPIRY_VERIFICATION_MS = 24 * 60 * 60 * 1000; // 24h
const TOKEN_EXPIRY_RESET_MS = 60 * 60 * 1000; // 1h

export function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, BCRYPT_COST);
}

export function verifyPassword(
  password: string,
  hash: string,
): Promise<boolean> {
  return bcrypt.compare(password, hash);
}

export function generateAccessToken(userId: string): string {
  return jwt.sign({ sub: userId }, process.env.JWT_SECRET!, {
    expiresIn: JWT_EXPIRY,
  });
}

export function verifyAccessToken(token: string): { sub: string } {
  return jwt.verify(token, process.env.JWT_SECRET!) as { sub: string };
}

export function generateRawToken(): string {
  return crypto.randomUUID();
}

export function hashToken(rawToken: string): string {
  return crypto.createHash("sha256").update(rawToken).digest("hex");
}

export function verificationExpiresAt(): Date {
  return new Date(Date.now() + TOKEN_EXPIRY_VERIFICATION_MS);
}

export function resetExpiresAt(): Date {
  return new Date(Date.now() + TOKEN_EXPIRY_RESET_MS);
}
```

### Task 7 — Create `src/server/services/emailService.ts`

Wraps Resend. Reads `RESEND_API_KEY`, `RESEND_FROM_EMAIL`, and `APP_URL` from env vars. Two exported functions: `sendVerificationEmail` and `sendPasswordResetEmail`.

```typescript
import { Resend } from "resend";

const resend = new Resend(process.env.RESEND_API_KEY);
const FROM = process.env.RESEND_FROM_EMAIL!;
const APP_URL = process.env.APP_URL ?? "http://localhost:5173";

export async function sendVerificationEmail(
  to: string,
  rawToken: string,
): Promise<void> {
  const link = `${APP_URL}/verify-email?token=${rawToken}`;
  await resend.emails.send({
    from: FROM,
    to,
    subject: "Verify your Finance Analyser email",
    html: `<p>Welcome to Finance Analyser!</p>
           <p><a href="${link}">Verify your email address</a></p>
           <p>This link expires in 24 hours. If you didn't create an account, ignore this email.</p>`,
  });
}

export async function sendPasswordResetEmail(
  to: string,
  rawToken: string,
): Promise<void> {
  const link = `${APP_URL}/reset-password?token=${rawToken}`;
  await resend.emails.send({
    from: FROM,
    to,
    subject: "Reset your Finance Analyser password",
    html: `<p>We received a request to reset your password.</p>
           <p><a href="${link}">Reset your password</a></p>
           <p>This link expires in 1 hour. If you didn't request this, ignore this email.</p>`,
  });
}
```

### Task 8 — Create `src/server/routes/auth.ts`

Express Router with all 6 POST routes. Imports from `src/db/index.ts` (Drizzle db instance), `authService.ts`, and `emailService.ts`.

**Route summary**:

| Route                              | Purpose                                         |
| ---------------------------------- | ----------------------------------------------- |
| POST /api/auth/register            | Validate, create user, send verification email  |
| POST /api/auth/verify-email        | Consume verification token, mark email_verified |
| POST /api/auth/resend-verification | Issue new verification token, invalidate old    |
| POST /api/auth/login               | Authenticate, return access token               |
| POST /api/auth/forgot-password     | Send reset email (identical response always)    |
| POST /api/auth/reset-password      | Consume reset token, update password            |

**Key implementation rules**:

- Email addresses normalised to lowercase before any DB read or write (`email.toLowerCase().trim()`)
- `POST /api/auth/forgot-password` and `POST /api/auth/resend-verification` always return 200 with a generic message — never reveal registration status
- `POST /api/auth/login` returns 401 for wrong email, wrong password, or unregistered email (generic — does not reveal which field was wrong)
- `POST /api/auth/login` returns 403 with `code: "EMAIL_NOT_VERIFIED"` for valid credentials on an unverified account
- Token hashing: `hashToken(rawToken)` before any DB write or comparison; raw token sent in email only
- Token expiry checked server-side before accepting any token
- Used tokens immediately nulled out: set `verificationToken = null`, `verificationTokenExpiresAt = null` (or reset equivalents) in the same DB update as the success action

See [contracts/api.md](contracts/api.md) for the complete request/response shapes.

### Task 9 — Create `src/server/middleware/requireAuth.ts`

JWT verification middleware for use by future protected API routes (not used by the auth routes themselves).

```typescript
import type { Request, Response, NextFunction } from "express";
import { verifyAccessToken } from "../services/authService.ts";

export function requireAuth(
  req: Request,
  res: Response,
  next: NextFunction,
): void {
  const header = req.headers.authorization;
  if (!header?.startsWith("Bearer ")) {
    res.status(401).json({ error: "Unauthorised", status: 401 });
    return;
  }
  try {
    const payload = verifyAccessToken(header.slice(7));
    (req as Request & { userId: string }).userId = payload.sub;
    next();
  } catch {
    res.status(401).json({ error: "Unauthorised", status: 401 });
  }
}
```

This middleware is not applied to any routes in this feature — it is the contract for future data endpoints. Register it per-route or per-router in subsequent features.

### Task 12 — Create `src/context/AuthContext.tsx`

Provides `AuthContext` with `token`, `user`, `login(token, user)`, and `logout()`. Reads from `sessionStorage` on mount. Writes to `sessionStorage` on login. Clears `sessionStorage` on logout.

```typescript
// State shape
interface AuthState {
  token: string | null;
  user: { id: string; email: string; displayName: string } | null;
}

// Context value
interface AuthContextValue extends AuthState {
  login: (token: string, user: AuthState["user"]) => void;
  logout: () => void;
}
```

Persist key: `"fa-auth-token"` in `sessionStorage` (JSON string containing `{ token, user }`). This key does not conflict with existing `pfa-v3-*` localStorage keys.

## Out of Scope (this feature)

- Token refresh / sliding session expiry
- Remember-me / persistent login across browser sessions
- Social login (Google OAuth, GitHub)
- Two-factor authentication
- Row-level security (Supabase RLS policies)
- Rate limiting on auth endpoints
- Account deletion
- Password complexity requirements beyond 8-character minimum
- Admin roles or user tiers
