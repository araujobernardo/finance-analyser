# Data Model: Email and Password Authentication

**Branch**: `010-email-password-auth` | **Date**: 2026-05-06

This feature adds authentication columns to the existing `users` table defined in feature 008. No new tables are introduced. A Drizzle migration (`0001_auth_tokens.sql`) applies the changes.

---

## Updated `users` Table

### Schema Change Summary

Feature 008 created the `users` table with 5 columns. This feature adds 5 new columns:

| New Column                    | Type        | Nullable | Default | Purpose                                         |
| ----------------------------- | ----------- | -------- | ------- | ----------------------------------------------- |
| email_verified                | boolean     | NO       | false   | Gate for sign-in; must be true to authenticate  |
| verification_token            | text        | YES      | NULL    | SHA-256 hash of the raw UUID verification token |
| verification_token_expires_at | timestamptz | YES      | NULL    | Token expiry (24h from issue time)              |
| reset_token                   | text        | YES      | NULL    | SHA-256 hash of the raw UUID reset token        |
| reset_token_expires_at        | timestamptz | YES      | NULL    | Token expiry (1h from issue time)               |

### Full Updated Drizzle Schema


### Column-by-Column Notes

**`email_verified`** (`boolean NOT NULL DEFAULT false`)

- Set to `true` only after successful email verification.
- Sign-in is blocked while this is `false`.
- Never reverts to `false` once set.

**`verification_token`** (`text NULLABLE`)

- Stores the SHA-256 hex digest of the raw UUID token sent to the user.
- Set to `NULL` after successful verification (token consumed).
- Replaced (set to new hash, old hash discarded) when a new verification email is requested.

**`verification_token_expires_at`** (`timestamptz NULLABLE`)

- Set to `now() + 24 hours` when a verification token is issued.
- Cleared (set to `NULL`) alongside `verification_token` after successful verification.
- Checked server-side: if `now() > verification_token_expires_at` the token is rejected.

**`reset_token`** (`text NULLABLE`)

- Stores the SHA-256 hex digest of the raw UUID reset token sent to the user.
- Set to `NULL` after the password reset is applied (token consumed).
- Replaced when a new reset email is requested (FR-022: new request invalidates old token).

**`reset_token_expires_at`** (`timestamptz NULLABLE`)

- Set to `now() + 1 hour` when a reset token is issued.
- Cleared alongside `reset_token` after successful password reset.

## Token Lifecycle

### Verification Token

```
ISSUE:
  raw_token = crypto.randomUUID()          // sent in email link
  stored_hash = sha256(raw_token)          // written to users.verification_token
  expires_at = now() + 24h                 // written to users.verification_token_expires_at

CONSUME (success):
  verification_token = NULL
  verification_token_expires_at = NULL
  email_verified = true

INVALIDATE (new token requested):
  stored_hash = sha256(new_raw_token)      // overwrites old hash
  expires_at = now() + 24h                 // new expiry
```

### Reset Token

```
ISSUE:
  raw_token = crypto.randomUUID()          // sent in email link
  stored_hash = sha256(raw_token)          // written to users.reset_token
  expires_at = now() + 1h                  // written to users.reset_token_expires_at

CONSUME (success):
  reset_token = NULL
  reset_token_expires_at = NULL
  hashed_password = bcrypt(new_password, 12)

INVALIDATE (new reset email requested):
  stored_hash = sha256(new_raw_token)      // overwrites old hash
  expires_at = now() + 1h                  // new expiry
```

## Migration: `0001_auth_tokens.sql`

Generated by `npm run db:generate` after updating `src/db/schema.ts`. The SQL will contain:

```sql
ALTER TABLE "users"
  ADD COLUMN "email_verified" boolean NOT NULL DEFAULT false,
  ADD COLUMN "verification_token" text,
  ADD COLUMN "verification_token_expires_at" timestamp with time zone,
  ADD COLUMN "reset_token" text,
  ADD COLUMN "reset_token_expires_at" timestamp with time zone;
```

This migration is:

- **Additive only** — no existing columns modified or removed.
- **Safe for existing rows** — `email_verified` defaults to `false`; all token columns default to `NULL`.
- **Idempotent** — `npm run db:migrate` applies it exactly once (tracked by `__drizzle_migrations` table).
