# Implementation Plan: Account Number-Based Categorisation

**Branch**: `006-account-number-categorisation` | **Date**: 2026-04-24 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/006-account-number-categorisation/spec.md`

## Summary

The system currently uses the account nickname (e.g. "Savings On Call") as the primary key when
grouping transactions by account. This causes accounts with the same name but different numbers
to be incorrectly merged. The fix is a two-line change in `parseAccountName` in `src/App.tsx`:
swap the key-priority so the account number takes precedence over the name, and update the
display format to show the full account number when both are available.

## Implementation Phases

### Phase 1 — Core Fix (non-UI)

**Story**: Fix `parseAccountName` to use account number as primary key

Files touched:

- `src/App.tsx` — change `short` priority and `baseDisplay` format

Acceptance: unit tests confirm that two CSV payloads with different account numbers but the same
nickname produce two distinct `short` values.

---

### Phase 2 — Display Update (UI)

**Story**: Show `"Name (number)"` in all account labels

This is a cosmetic consequence of Phase 1 — the `display` value returned by `parseAccountName`
already changes format, so all UI surfaces that render `account` or `accountList[x].display`
will automatically pick up the new label with no further changes.

Visual verification: the sidebar, account selector, and settings page all show
"Savings On Call (0549256-53)" and "Savings On Call (0549256-50)" as distinct entries.

---

### Phase 3 — Tests

**Story**: Regression tests for `parseAccountName`

Unit tests (Vitest) covering:

1. Two CSV payloads with same name, different numbers → two different `short` values
2. CSV with number + name → `short` is the number, `display` is `"Name (number)"`
3. CSV with name only (no number) → `short` is the name, `display` is the name
4. CSV with number only (no name) → `short` is the number, `display` is the number
5. CSV with no recognisable metadata → falls back to `"Main"` / `"Main Account"`
