# Implementation Plan: Manual Transfer Flagging

**Branch**: `003-manual-transfer-flagging` | **Date**: 2026-04-24 | **Spec**: [spec.md](spec.md)

---

## Summary

Add a user-initiated transfer-flagging workflow to the Transactions page. Clicking any non-transfer row opens a candidate picker that filters same-day, same-absolute-amount transactions; selecting a candidate flags both as a transfer pair. Clicking any visible transfer row opens an un-flag confirmation that restores each transaction's prior category. All state changes are persisted to localStorage via the existing `onBulkCategoryChange` callback path.

## Data Model Change

### `PfaTxn` (extended)

Add one optional field to the existing interface in `src/types/pfa.ts`:

```typescript
preFlagCategory?: string | null;
```

**Semantics**:

- Set when `isTransfer` is set to `true` via manual flagging. Contains the `category` value the transaction held immediately before flagging.
- Cleared (`undefined`) when `isTransfer` is set back to `false` via un-flagging.
- Never set by auto-detection (`detectTransfers` in `App.tsx` is not modified).
- Auto-detected transfers have `preFlagCategory` absent/undefined; on un-flag these revert to `category: null`.

**Migration**: No migration needed. Existing persisted records simply lack the field (treated as `undefined` by TypeScript's optional field semantics).

### Phase 2 — Flag / Un-flag Logic in TransactionsPage

**Goal**: Implement the core state machine and business logic inside `TransactionsPage`.

**File**: `src/pages/TransactionsPage.tsx`

**New state**:

- `flagMode: { initiatingId: string } | null` — when non-null, the app is in "flag as transfer" mode
- `unflagTarget: { txnId: string } | null` — when non-null, the un-flag confirmation is shown

**Candidate derivation** (computed from `flagMode` and `txns`):

```
candidates = txns where:
  t.id !== initiatingId
  AND t.date === initiatingTxn.date
  AND Math.abs(t.amount) === Math.abs(initiatingTxn.amount)
  AND !t.isTransfer
```

**Flag action** (`handleFlag(initiatingId, candidateId)`):

1. Find both transactions.
2. For each: set `preFlagCategory = current category`, `category = "Savings & Transfers"`, `isTransfer = true`.
3. Call `onBulkCategoryChange(updatedTxns)`.
4. Clear `flagMode`.

**Un-flag action** (`handleUnflag(txnId)`):

1. Find the transaction and its partner (another `isTransfer` txn with same `date` and `Math.abs(amount)`, excluding itself).
2. For each found transaction: set `isTransfer = false`, `category = preFlagCategory ?? null`, `preFlagCategory = undefined`.
3. Call `onBulkCategoryChange(updatedTxns)`.
4. Clear `unflagTarget`.

**Row click routing**:

- Non-transfer row: if `flagMode` is null → set `flagMode = { initiatingId: t.id }`; if `flagMode` is active and `t.id` is a candidate → call `handleFlag(flagMode.initiatingId, t.id)`.
- Transfer row: set `unflagTarget = { txnId: t.id }` to open confirmation.
- Escape / click outside: clear both `flagMode` and `unflagTarget`.

**Acceptance**: All acceptance criteria in spec.md US1 and US2 pass. Existing bulk-category-change behaviour is unchanged.

### Phase 3 — UI Presentation

**Goal**: Visual feedback for flagging mode and un-flag confirmation.

**File**: `src/pages/TransactionsPage.tsx` + `src/pages/TransactionsPage.css`

**Flagging mode UX**:

- Initiating row is highlighted with an accent-coloured left border and a subtle background tint.
- Candidate rows are highlighted with a dashed accent border.
- Non-candidate, non-initiating rows are dimmed (50% opacity).
- A dismissable banner at the top of the table reads: "Select the matching transaction to complete the transfer pair — or press Escape to cancel."
- If no candidates exist, the banner reads: "No matching transactions found for this day and amount."

**Un-flag confirmation UX**:

- An inline confirmation row or small panel appears below the clicked transfer row (or at top of table as an overlay).
- Text: "Un-flag this transfer pair? Both transactions will revert to their previous categories."
- Two buttons: "Confirm" (calls `handleUnflag`) and "Cancel" (clears `unflagTarget`).

All new styles use `var(--*)` CSS tokens only — no hardcoded hex values.

**Acceptance**: Visual states match description. No hardcoded colours.

### Phase 4 — Tests

**Goal**: Unit tests covering flag/unflag logic.

**File**: `src/pages/TransactionsPage.test.tsx`

**Required test cases**:

1. `handleFlag` sets `isTransfer: true` and `category: "Savings & Transfers"` on both transactions.
2. `handleFlag` stores the original categories in `preFlagCategory` for both transactions.
3. `handleUnflag` sets `isTransfer: false` and restores `category` from `preFlagCategory` for both transactions.
4. `handleUnflag` on auto-detected transfers (no `preFlagCategory`) sets `category: null`.
5. Candidate filtering returns only same-day same-amount non-transfer transactions, excluding the initiating transaction.
6. Candidate filtering returns an empty array when no matching transactions exist.

**Acceptance**: All 6 test cases pass. No existing tests regress.
